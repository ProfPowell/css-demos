02 - Inline Elements

Inline elements are boxes, like block elements, but their default behavior is to occupy and define only the space of their child content, not the whole vertical and horizontal 'row' space outside of it.

While block elements can contain other block and inline elements, inline elements should only contain other inline elements. A great example is a text link. The <a> tag's semantics informs the device that this element is interactable, and when clicked, the device should read its action attribute and change the page to the new attribute value (url).

Typically, a link is found inside a paragraph surrounded by other text and links. If a link was a block element, then the text before and after it would be placed on different lines. Instead, since it is an inline element, it retains its normal flow in the paragraph's text, yet offers a wrapper box to provide semantics and to hook onto for CSS and JavaScript.

Move your mouse over the area to the right of the text not inside the dashed border. Your mouse cursor icon will not change, and clicking will not active the link. Remember, inline elements just occupy the space of their content.

If we make the outer <a> tag a block element, you'll notice that the entire horizontal space is filled with the element's dimensions, so clicking in the empty whitespace will activate the text link.

Note: While you can probably envision instances where you might want to make a block element clickable, it is important to know this historical relationship between block and inline. Most modern browsers tend to correctly interpret block elements inside inline elements, however from time-to-time rendering issues do crop up.

In general, it is best to use CSS to apply block behavior to inline elements inside other inline elements than actually adding a semantic block element inside a parent inline element.

Therefore it usually more appropriate to do this:

<div>
  <a>...</a>
</div>

instead of this:

<a>
  <div>...</div>
</a>

The most common inline elements are:

  • <span> tags are the generic inline equivalent to the block <div>. It provides no semantic value. You cannot infer the inner content's meaning from the tag itsef.
  • <em> tags convey normal emphasis. Devices typically render the containing text as italic, but it is important to remember the semantic value the tag provides (emphasis), over its presentational interpretation (italic). The device could easily choose to render these elements as underlined text in the future. However, its semantic value wouldn't change.
  • <strong> tags convey stronger emphasis. Devices typically render the containing text as bold, but it is important to remember the semantic value the tag provides (stronger emphasis than a <em> tag), over its presentational interpretation (bold). Like the em tag, the device could easily choose to render these elements as underlined text in the future. However, its semantic value wouldn't change.
  • <a> tags create hyperlinks to other pages or sites. It expects an action attribute, href="https://www.google.com", whose value is a valid url, or anchor tag:
    <a href="https://www.google.com">...</a>

Notes: Devices tend to add their own styling to specific tags. Because this can be inconsistent across different browsers, you shouldn't rely on this alone. If the styling is important, it should be applied with CSS.

Make sure you use the semantic tag instead of its older (deprecated) presentational variant, where applicable. So <strong> instead of <b>, and <em> instead of <i>.

With CSS, you can turn any element into an inline element with the display property: display: inline;
